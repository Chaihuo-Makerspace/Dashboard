# 多 CC 窗口并行开发 Git 工作流设计

**日期**：2026-04-17
**范围**：git 工作流规范、项目记录文档体系、全局 CLAUDE.md 规则

---

## 1. 目标

在单人多 Claude Code 窗口并行开发场景下，建立一套可重复执行的 git 工作流，使得：

- 每个 CC 窗口独立在自己的分支上工作，互不干扰
- 任务完成后可安全合并回 main（遇冲突停下等人决策）
- 新打开的 CC 窗口无需人工介绍，读 CLAUDE.md 即可了解项目现状
- GitHub 推送由用户主动触发，CC 不自动推送

---

## 2. 整体架构

```
~/.claude/CLAUDE.md          ← 全局规则（所有项目通用）
项目根/CLAUDE.md             ← 项目上下文（每个项目独立）
```

**工作流循环：**

```
新 CC 窗口启动
  → 读 ~/.claude/CLAUDE.md（了解工作流规则）
  → 读 项目/CLAUDE.md（了解项目现状）
  → 从 main 创建新分支（feature/xxx 或 fix/xxx）
  → 完成任务，提交
  → 告知用户「任务完成，可以合并」
  → 用户确认 → CC 执行 merge commit 到 main
  → CC 更新项目 CLAUDE.md 变更日志
  → 等待用户的 GitHub 推送指令
```

---

## 3. 分支命名 & 提交规范

**分支命名：**

| 前缀 | 用途 |
|------|------|
| `feature/` | 新功能 |
| `fix/` | 缺陷修复 |
| `refactor/` | 重构 |
| `docs/` | 文档 |

**提交信息：**

```
feat: 描述
fix: 描述
docs: 描述
chore: 描述
refactor: 描述
```

---

## 4. 合并策略

- 使用 `merge commit`（`--no-ff`），保留完整分支历史
- 合并信息格式：`merge: <branch-name> into main`
- **遇到冲突：立即停止，列出冲突文件，等待用户决策，禁止自动解决**
- 合并完成后更新项目 CLAUDE.md 变更日志
- 分支合并后保留不删，方便回溯

**合并命令：**

```bash
git checkout main
git merge --no-ff <branch-name> -m "merge: <branch-name> into main"
```

---

## 5. GitHub 推送规则

- CC 不主动推送，必须等用户明确说「推送」再执行
- 推送命令：`git push origin main`

---

## 6. 项目 CLAUDE.md 模板

```markdown
# 项目名称

## 项目简介
一句话描述这个项目是什么、做什么用。

## 技术栈
- 列出主要技术

## 目录结构
关键文件说明（只列重要的）

## 当前状态
- 活跃分支：列出正在进行的分支及其任务
- 已完成功能：简要列表

## 重要约定
- 特定于本项目的注意事项
- 已知问题或限制

## 变更日志
| 日期 | 分支 | 说明 |
|------|------|------|
| YYYY-MM-DD | branch-name | 做了什么 |
```

**更新时机：**

- 每次 merge 完成后，追加一行变更日志
- 新功能上线时更新「当前状态」
- 发现重要约定时更新「重要约定」

---

## 7. 全局 ~/.claude/CLAUDE.md 结构

内容是工作流规则，不含项目细节，对所有项目生效。详见实施计划中的完整内容。
